    Deployment#

    A Deployment is a collection of pods defined by a template. For the simplest deployments all you need to configure is some metadata about which pods it should contain and how many instances of each pod should be available.

    When this is deployed, the deployment will create a ReplicaSet which then creates the required number of Pods. When you update the application, the default update strategy is a RollingUpdate, so during the update a new ReplicaSet will be created for the new version of the application, and then as Pods are successfully started in that new ReplicaSet, the previous ReplicaSet will scale down until you've migrated completely to the new version.

    ---
    apiVersion: apps/v1
    kind: Deployment
    metadata:
    name: hello-world
    labels:
    app: hello-world
    spec:
    # How many pods should be running
    replicas: 10
    # How do we find pods that belong to this deployment? The
    # deployment uses the matchLabels to determine which pods are part
    # of this deployment.
    selector: { matchLabels: { app: hello-world } }
    # The template defines what the pods in the deployment look like.
    template:
    # The labels in this metadata block get applied to all of the
    # launched pods, they match the labels in the selector above so
    # that all of the pods created from this deployment will be
    # associated with it.
    metadata: { labels: { app: hello-world } }
    # The spec defines what containers should be running the pods
    spec:
    containers:
    - name: hello-world-rest-server
    image: hellow-world-api:1.0
    ports:
    - containerPort: 80

    Deployments are best suited for stateless applications, where pods can be replaced at any time without breaking things, but you can get a persistent volume if your application needs to store some state.

    ---
    apiVersion: apps/v1beta1
    kind: Deployment
    metadata: { name: counter }
    spec:
    replicas: 1
    template: { metadata: { labels: { app: counter } } }
    spec:
    containers:
    - name: counter
    image: "kahootali/counter:1.0"
    volumeMounts:
    # The name in the volumeMount refers to the name of the volume
    # in the following `volumes` section.
    - name: counter
    mountPath: /app/
    volumes:
    - name: counter
    persistentVolumeClaim:
    # This claimName refers to the metadata.name in the
    # PersistentVolumeClaim defined in the next section
    claimName: counter
    ---
    apiVersion: v1
    kind: PersistentVolumeClaim
    metadata: { name: counter }
    spec:
    accessModes: [ ReadWriteMany ]
    resources:
    # This is how the size of this persistent volume is determined
    requests:
    storage: 50Mi
    storageClassName: default

    In this example, the persistent volume is mounted read/write on every pod in the deployment.
